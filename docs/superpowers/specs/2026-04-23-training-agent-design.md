# 岗位培训 Agent 设计规格

**日期：** 2026-04-23
**状态：** 已确认

---

## 1. 背景与目标

### 1.1 项目定位

基于 gbrain 知识库系统，扩展培训插件模块，为企业提供**新员工入职培训**和**岗位技能提升**一体化解决方案。

### 1.2 核心价值

- **企业微信/钉钉集成**：员工在日常 IM 工具中完成学习，无需切换系统
- **AI 课件生成**：从 gbrain 知识库自动提取内容，生成结构化培训课件
- **任务驱动学习**：HR/管理员下发任务，员工完成学习 + 测验，记录全程数据
- **三方数据看板**：管理员、HR、员工各自看到对应维度的数据

### 1.3 使用场景

| 场景 | 描述 |
|------|------|
| 新员工入职培训 | 帮助新员工快速了解公司制度、流程、岗位技能 |
| 岗位技能提升 | 针对现有员工技能短板，推荐学习内容并验证掌握程度 |

### 2.1 整体架构

```
┌─────────────────────────────────────────────────────────┐
│                     企业微信 / 钉钉                        │
│              (员工交互入口，消息推送)                       │
└─────────────────────┬───────────────────────────────────┘
                      │ 直接 API 调用（REST Webhook 回调）
┌─────────────────────▼───────────────────────────────────┐
│                  gbrain 主服务                            │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐      │
│  │   知识库     │  │   知识图谱   │  │   语义搜索   │      │
│  │  (gbrain)   │  │  (gbrain)   │  │  (gbrain)   │      │
│  └─────────────┘  └─────────────┘  └─────────────┘      │
└─────────────────────┬───────────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────────┐
│              training 插件模块 (gbrain/plugins/training/) │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │
│  │  课件生成器  │  │  学习状态机  │  │  事件总线    │     │
│  │(CourseGen)  │  │(StateMachine)│ │(EventBus)   │     │
│  └─────────────┘  └─────────────┘  └─────────────┘     │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐     │
│  │  任务推送器  │  │  考核引擎    │  │  数据看板    │     │
│  │(TaskPusher) │  │(QuizEngine) │  │(Dashboard)  │     │
│  └─────────────┘  └─────────────┘  └─────────────┘     │
└─────────────────────────────────────────────────────────┘
```

### 2.2 插件架构

training 模块以插件形式挂载在 gbrain 主服务下，独立模块、独立数据存储，通过事件总线通信。

**模块职责：**

| 模块 | 职责 |
|------|------|
| CourseGen（课件生成器） | 从 gbrain 知识库提取内容，AI 生成结构化课件 + 测验题 |
| StateMachine（学习状态机） | 管理每个员工-任务对的学习进度状态 |
| EventBus（事件总线） | 模块间通信中枢，所有状态变更通过事件驱动 |
| TaskPusher（任务推送器） | 消费事件，通过企微/钉钉 API 推送消息 |
| QuizEngine（考核引擎） | 评分、记录测验成绩、决定是否通过 |
| Dashboard（数据看板） | 聚合数据，提供管理员/HR/员工三个视角的报表 |

### 2.3 技术选型

| 组件 | 技术选型 | 说明 |
|------|----------|------|
| 插件框架 | gbrain 目录结构 | `gbrain/plugins/training/` |
| 消息总线 | Redis Pub/Sub | 轻量、已在 .venv 中可安装 |
| 任务队列 | Redis Queue (RQ) | 适合定时任务、异步任务下发 |
| 企业微信 API | 直接调用 REST API | 官方文档完善，WebHook 回调 |
| Web 页面 | FastAPI + Jinja2 模板 | 复用 Python 技术栈 |
| 数据存储 | PostgreSQL | 结构化数据（员工、任务、进度） |

## 3. 数据模型

### 3.1 核心实体

**员工 (Employee)**
- id, name, department, position, join_date
- role: admin | hr | employee
- wecom_openid / dingtalk_userid（企微/钉钉用户 ID）

**培训任务 (TrainingTask)**
- id, title, description, task_type: onboarding | skill
- content_source: 来自 gbrain 知识库的页面 ID 列表
- quiz_items: 测验题列表（JSON）
- deadline, priority
- status: draft | published | archived
- created_by, created_at

**学习进度 (LearningProgress)**
- id, employee_id, task_id
- state: not_started | learning | quiz_failed | completed | mastered
- started_at, completed_at
- quiz_score, quiz_attempts
- events: 状态变更历史

**事件记录 (Event)**
- id, timestamp, event_type
- source: employee_id | system | hr_action
- payload: 事件相关数据（JSON）
- consumed: bool

### 3.2 状态机定义

```
not_started
    │ 员工打开任务
    ▼
learning
    │ 员工提交测验
    ▼
quiz_failed (分数 < 60%) ──→ [补考机会 2 次] ──→ learning
    │ 补考通过
    ▼
completed (分数 >= 60%)
    │ 自动或手动标记为已掌握
    ▼
mastered
```

### 3.3 事件类型

| 事件 | 触发时机 | 下游处理 |
|------|----------|----------|
| task_created | 管理员创建任务 | 推送任务通知给员工 |
| task_assigned | 任务分配给员工 | 更新进度状态为 not_started |
| learning_started | 员工打开任务 | 记录开始时间 |
| quiz_submitted | 员工提交测验 | 考核引擎评分，更新状态 |
| quiz_passed | 测验及格 | 推送恭喜通知，更新进度 |
| quiz_failed | 测验不及格 | 推送补考通知 |
| task_completed | 任务完成（分数 >= 60%） | 更新进度，推送完成确认 |
| task_mastered | 任务标记为已掌握 | 更新进度，推送勋章/积分 |

## 4. 核心交互流程

### 4.1 培训任务下发（管理员/HR 触发）

```
HR 在 Web 页面创建培训任务
    │
    ▼
课件生成器 ──→ 从 gbrain 知识库提取相关内容
    │           AI 生成课件 + 测验题
    ▼
任务入库 + 事件发布 (task_created)
    │
    ▼
事件总线 → TaskPusher 消费
    │
    ▼
企微/钉钉 API 调用 → 员工收到任务通知卡片
```

### 4.2 员工学习闭环

```
员工点击企微/钉钉中的任务卡片
    │
    ▼
跳转至学习页面（Web 端）
    │   - 阅读课件内容
    │   - 完成随堂测验
    ▼
提交测验 → QuizEngine 评分
    │
    ├── 及格（>= 60%）→ 状态机：completed
    │                        事件：task_completed
    │                        推送：恭喜完成
    │
    └── 不及格（< 60%）→ 状态机：quiz_failed
                             补考机会 -1
                             推送：补考通知
                         → 员工重新学习 → 再次提交测验
```

### 4.3 管理/HR 数据查看

```
HR/管理员登录数据看板
    │
    ▼
Dashboard 聚合数据
    │
    ├── 整体覆盖率统计（已完成/总人数）
    ├── 各部门/岗位完成率
    ├── 测验正确率分布（薄弱知识点）
    ├── 未完成员工列表（催学用）
    └── 导出考核报告（Excel/PDF）
```

## 5. 企业微信/钉钉集成

### 5.1 接入方式

直接 API 集成，不依赖 MCP：

- **企业微信**：使用企业微信「应用消息」API，主动推送消息给员工；接收员工消息通过 WebHook 回调
- **钉钉**：使用钉钉「机器人」API，主动推送消息；接收消息通过 Stream 模式或回调

### 5.2 消息类型

| 消息类型 | 触发场景 |
|----------|----------|
| 任务通知卡片 | 新任务下发（包含标题、描述、链接、截止时间） |
| 测验提醒 | 任务截止前 X 天自动提醒 |
| 补考通知 | 测验不及格后立即通知 |
| 完成确认 | 任务完成后推送确认消息 |
| 催学通知 | 任务即将到期或管理员手动触发 |

### 5.3 回调处理

员工在企微/钉钉中点击卡片跳转到 Web 学习页面（URL 携带加密参数），Web 页面完成身份验证后展示内容。

## 6. 错误处理

| 场景 | 处理方式 |
|------|----------|
| 企微/钉钉 API 调用失败 | 重试 3 次 + 记录失败事件，人工告警 |
| 员工网络断开 | 学习进度本地缓存（LocalStorage），重新上线后同步 |
| AI 生成的课件质量不佳 | 管理员可人工编辑，保留版本记录 |
| 测验泄题风险 | 题目随机打乱，选项随机排序，企微侧加水印 |
| 重复提交测验 | 幂等处理，以最后一次提交为准 |
